# Kernel Function Support Extension

To extend function support in the PoCL/CuPBoP compiler, one of the simplest approaches is to use PoCL’s built-in libraries. Users can extend Clang with user-defined functions that are implemented in PoCL/CuPBoP’s built-in library.

Supporting easy extension of new functions is particularly important for GPU compilers, because GPU variants may introduce new features and ISA extensions (e.g., tensor-core instructions). In addition, some low-level capabilities must be exposed at the user level to make new hardware features usable (e.g., CUDA WMMA APIs for tensor cores).

In this tutorial, we explain how to extend user-level kernel function support by adding support for the `vx_shuffle_up` instruction. The `vx_shuffle_up` instruction shifts `value` up by `bval` lanes within each sub-group; out-of-range lanes keep their original value. The instruction takes four inputs: `value`, `bval`, `cval`, and `mask` and we can use a signature that matches the hardware instruction in this case.

```c
int vx_shfl_up(size_t value, int bval, int cval, int mask);
```

## 1. Extending the PoCL Compiler

### 1.1 Extend the declarations shipped with Clang

PoCL provides an include file for built-in function declarations (`include/_clang_opencl.h`). Add a new declaration for `shuffle_up`. 

```c
int pocl_vx_shfl_up(int value, int bval, int cval, int mask);
```

### 1.2 Extend the built-in library

Under `lib/kernel/vortex`, VOLT provides the built-in library for Vortex. Add a new file called `vortex_extension.c`:

```bash
cd lib/kernel/vortex
touch vortex_extension.c
```

Then implement `pocl_vx_shfl_up` using the [Vortex kernel library](https://github.com/vortexgpgpu/vortex/blob/master/kernel/include/vx_intrinsics.h):

```c
#include <vx_intrinsics.h>

int pocl_vx_shfl_up(int value, int bval, int cval, int mask) {
  return vx_shfl_up(value, bval, cval, mask);
}
```

### 1.2. Using the function in an OpenCL kernel

Below is a minimal OpenCL kernel example that calls `pocl_vx_shfl_up` inside a work-group. The example shifts values “up” by one lane within the sub-group/warp. Lanes that are out of range (e.g., lane 0 for an up-shift) keep their original value.

```c
__kernel void shfl_up_u32(__global const int* in,
                          __global int* out,
                          const int n) {
  const int gid  = (int)get_global_id(0);
  const int lid  = (int)get_local_id(0);

  if (gid >= n) return;

  // Per-thread input value.
  const int value = in[gid];

  // Shift value up by 1 lane in the sub-group/warp.
  // - bval: delta (1 lane)
  // - cval: implementation-defined (often used as clamp/control); use 0 for a basic example
  // - mask: active-lane mask; use all-ones for “all lanes active”
  const int bval = 1;
  const int cval = 0;
  const int mask = -1;

  const int shuffled = pocl_vx_shfl_up(value, bval, cval, mask);

  // Optional: make the boundary behavior explicit (if desired).
  // If lane 0 should keep its own value for an up-shift:
  // const int result = (lid == 0) ? value : shuffled;

  out[gid] = shuffled;
}
```

## 2. Extending the CuPBoP Compiler

To support shuffle up fuction, we will support CUDA `__reduce_add_sync` by lowering the NVVM intrinsic `llvm.nvvm.redux.sync.add` to our CuPBoP runtime function. We will use a simple implementation that utilizes existing CuPBoP vote shuffle function to perform reduction.

### 2.1 Implement runtime helper
First, implement `__my_reduce_add_sync` at `runtime/src/vortex/kernel/cudaKernelImpl.cpp` and insert its declaration at `runtime/include/cudaKernelImpl.h`. 

```C++
#include <vx_intrinsics.h>
...
// cudaKernelImpl.cpp
__attribute__((always_inline))
int __my_reduce_add_sync(unsigned mask, int val) {
    for (int offset = 16; offset > 0; offset >>= 1) {
        int other = vx_shfl_up(mask, val, offset, 0);
        val += other;
    }
    return val;
}

// cudaKernelImpl.h
__attribute__((always_inline))
int __my_reduce_add_sync(unsigned mask, int val);
```

### 2.2 Lower NVVM intrinsic to CuPBoP runtime function

Next, we need to identify the corresponding CUDA intrinsic `@llvm.nvvm.redux.sync.add` in LLVM IR and replace its function call with our custom function. Note that although `__reduce_add_sync` can also be matched as a CUDA function in LLVM IR, we should target the actual intrinsics, since the wrapper function can always be inlined away by the compiler. At `compilation/Kerneltranslation/src/warp_func.cpp`, under `replace_built_in_function`, add the following lines:

```C++
            else if (func_name == "llvm.nvvm.redux.sync.add") {
              Call->getCalledFunction()->setName("__my_reduce_add_sync");
            }
``` 
This is a simple name matching but can be extended to support variants if needed.

### 2.3 Test the function in a CUDA kernel
Finally, we can implement a simple kernel to test `__reduce_add_sync` support.

```C++
__global__ void block_reduce_kernel(int *data, int *result, int n) {
    int tid = threadIdx.x + blockIdx.x * blockDim.x;
    int val = data[tid];
    int sum = __reduce_add_sync(0xFFFFFFFF, val);
    if (threadIdx.x == 0) {
        atomicAdd(result, sum);
    }
}
```
