# Extend Memory Lowering in CuPBoP

In this tutorial, we use shared-memory support as an example to show how memory-related extensions are implemented in Volt (**CuPBoP-Vortex**), and how users can extend the selection and validation workflow for alternative behaviors with minimal disruption to existing CUDA applications. CUDA programs typically assume a specific shared-memory abstraction: a per-CTA scratchpad with fast, low-contention access. On open GPU targets, however, there is no single “**best**” implementation. The optimal mapping depends on the target memory hierarchy and, in Volt’s case, on the chosen Vortex soft-core configuration under fixed hardware resources. In practice, some applications benefit more from a larger cache hierarchy, while others benefit more from explicit local memory. As a result, CUDA shared memory must be reinterpreted under the target memory model, and the implementation becomes a policy choice rather than a fixed rule.

CuPBoP therefore treats shared memory as a swappable lowering strategy. Depending on the kernel’s access pattern and the available memory hierarchy, shared memory can be cast to per-core local memory (to reduce contention and improve locality) or mapped to global memory (to leverage caching and improve throughput). Volt exposes this choice as a runtime-configurable policy so that users can quickly compare outcomes and select the best behavior per application.

In this tutorial, we walk through two shared-memory lowering strategies—shared-to-global casting and shared-to-local casting—and show where and how to modify the mapping policy when needed.

First, Volt allows users to select the shared-memory casting policy via an environment variable:

```bash
export VORTEX_LOCALMEM_FLAG=0/1  # 1 = use local memory, 0 = cast shared memory to global memory
```

If you are unsure which policy best matches your application, a practical approach is to run both configurations and compare performance. Next, we examine how these two modes are implemented inside CuPBoP-Vortex.

## 1. Extending Kernel Translator in CuPBoP_Vortex

Inside `KernelTration/src/memory_hierarchy.cpp` we have two functions, 
```c
void mem_share2global_sche_2(llvm::Module *M);
void mem_share2local(llvm::Module *M);
``` 

At a high level, both functions iterate over module-level global variables and identify shared-memory globals by checking whether addressSpace == 3. The key difference is what memory object they replace the shared global with, and how they ensure per-CTA isolation.

First, mem_share2global_sche_2 replaces each shared global with a newly created module global variable in address space 1. As shown below, it allocates a replacement global in AS=1 and then rewrites uses so that each CTA indexes into its own slice. The CTA identifier (block ID) is loaded at runtime, ensuring that different CUDA blocks access disjoint regions of the new global object.

```c
auto createNoTLSGlobal = [&](Type *Ty, StringRef Name) -> GlobalVariable* {
  auto *Init = isa<ArrayType>(Ty) ? (Constant*)ConstantAggregateZero::get(Ty)
                                  : (Constant*)Constant::getNullValue(Ty);
  auto *GV = new GlobalVariable(
      *M, Ty, /*isConst=*/false,
      GlobalValue::ExternalLinkage, Init, Name,
      /*InsertBefore=*/nullptr,
      GlobalValue::NotThreadLocal, /*AddressSpace=*/1);
  GV->setThreadLocal(false);
  GV->setThreadLocalMode(GlobalVariable::NotThreadLocal);
  return GV;
};
```

In contrast, mem_share2local maps shared memory to Vortex per-core local memory. When a shared-memory variable is detected, the transformation inserts runtime allocation logic so that each CTA’s leader thread (e.g., tid == 0) allocates a local-memory region using the Vortex local-memory allocation intrinsic. The pass then rewrites shared-memory accesses to use the allocated base address, enabling accesses to be served from Vortex local memory.

```c
FunctionCallee vxAlloc = M->getOrInsertFunction(
    "vx_local_alloc", FunctionType::get(I8Ptr, {I32}, false));
```

These memory-casting transformations are invoked from the init_block function in `KernelTranslation/src/init.cpp`. When the user sets the **global environment variable** `VORTEX_LOCALMEM_FLAG`, CuPBoP-Vortex uses this runtime policy to select the appropriate lowering pass under the hood: `mem_share2global_sche_2` (cast shared memory to global) or `mem_share2local` (map shared memory to local memory). As a result, users can evaluate both shared-memory mappings without modifying the CUDA source code, making it straightforward to experiment with alternative memory hierarchies and validate which mapping is more effective for a given kernel and hardware configuration.

## Implementing an Additional Memory-Mapping Policy

To extend this mechanism with an additional memory-mapping policy, add a new lowering function (for example, `mem_share2global_and_local_sche_2`) in `KernelTranslation/src/memory_hierarchy.cpp`, then wire it into the policy selection logic in `KernelTranslation/src/init.cpp` as shown below:

```c
if (schedule == 0) {
  mem_share2global(M);
} else if (schedule == 2) {
  if (local_mem_use == 1) {
    mem_share2local(M);
  }
  // Newly added function
  // else if (local_and_global_mem == 1) {
  //   mem_share2global_and_local_sche_2(M);
  // }
  else {
    mem_share2global_sche_2(M);
  }
}
```
