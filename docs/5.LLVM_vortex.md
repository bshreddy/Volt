# LLVM-Vortex 

## 1. LLVM-Vortex Structure

Our key LLVM extensions are under `llvm/lib/Target/RISCV`.
Especially, the following files are particularly important.

- `RISCVTargetMachine.cpp`: All lowering sequences and compiler option optimization sequences are managed here. 
- `VortexBranchDivergence.cpp`: Vortex divergence management.
- `RISCVInstrInfoVX.td`: Vortex ISA table and intrinsics.

Our compiler focus on divergence management with uniformity analysis, CFG optimization and transformation passes, and divergence management function insertion passes.

## 2. Uniformity Analysis

Uniformity analysis is the first stage of divergence management and determines whether a program variable maintains the same value across all threads. Uniformity analysis classifies each value as either **uniform** or **divergent**, 

- **Uniform:** the value is identical across all threads in a warp/wavefront. If a predicate is **uniform**, all threads follow the same path, enabling efficient SIMT execution.
- **Divergent:** the value may differ across all threads. If a predicate is **divergent**, threads may take different paths, requiring explicit divergence handling (e.g., divergence tracking plus reconvergence management).

This distinction is foundational because the semantics and performance of control flow (e.g., `branch`) depend on the uniformity of the controlling predicate:

Since RISC-V is CPU-oriented, the upstream `llvm-riscv` back-end does not model SIMT branch divergence by default. 
Our uniformity analysis is built on top of LLVM’s uniform analysis by extending the abstract interface (the RISC-V TTI interface). This interface has two important virtual functions:

- `isAlwaysUniform`: return `true` when the value is always-uniform, such as constants.
- `isSourceOfDivergence`: return `true` when the value is a source of divergence, such as thread identifiers.

Our uniformity analysis implements these two virtual functions using GPU program semantics and hardware characteristics.

### Divergence Tracker

To find and track sources of divergence, VOLT defines two structures, `DivergenceTracker` and `DivergenceInfo`, for tracking divergence. This tracker marks values as divergent under the following conditions:

1. Function arguments
   - Conservatively treated as **potentially divergent** unless explicitly marked uniform.

2. Atomic operations
   - Treated as **divergent** because results may differ across threads due to inter-thread ordering effects.

3. Function return values
   - Conservatively treated as divergent unless proven/marked uniform via `isAlwaysUniform`.
   - This includes intrinsics such as `threadlocal_address` used to access TLS variables.

4. Machine-level CSRs access operation
    - Access machine Control and Status Registers such as the number of hardware threads in cores.
    - Access program CSRs such as core id. 


### Annotation Analysis

To identify always-uniform values based on metadata, VOLT uses annotation analysis implemented as `UniformAnnotationPass`. 

1. Intrinsic-based analysis: inspects global variables, constant data variables, and stack variables using front-end compiler metadata. 
    - For stack variables, `store`/`load` operations are conservatively handled such that accesses to stack variables are treated as **uniform**.

2. User-provided annotations
   - Uniformity can also be derived from user guidance. The analysis consumes and cross-checks annotations (e.g., `__UNIFORM__`).

### Function Argument Analysis

Some function arguments can be **uniform** values. VOLT also analyzes the program to determine whether each function argument is uniform. The analysis builds a call graph and runs function-level analysis in reverse post-order to determine the uniformity of each function argument and return value. See `VortexDivergenceArguments` for details. 

## 3. Optimization and Transformation

VOLT uses IR- and CFG-level transformations to simplify control flow, to structurize the control-flow graph for SIMT divergence management, lower divergence operations to branches, and selectively reconstruct the CFG to reduce predicate-computation overheads introduced by linearization.

1. CFG simplification: applies a normalization phase followed by LLVM canonicalization and CFG cleanup. 

2. Control-Flow Structurization: structurizes the CFG, because structured control flow yields stable reconvergence points for thread-mask–based execution.

3. Divergence Operation Lowering: normalizes the divergence operations by rewriting them into equivalent branch-based control flow. See `VortexBranchDivergence0` for details. 
   - Investigate divergence `select`, `min`, `max`, and integer conditional operations.
   - Remove unreachable blocks

4. CFG Reconstruction
   - Structurization and linearization often introduce predicate computations. When unstructured regions are deeply nested, predicate computation can become expensive and dominate the transformation overhead.
   - Given an unstructured region with complex control dependence, duplicates selected nodes to reduce predicate complexity.
   - See `VortexReconstructionPass.cpp` for details. 

## 4. Divergence Management Function Insertion

After reconstructing the CFG and analyzing divergent and uniform variables, VOLT inserts divergence and reconvergence points based on the analysis result. See `VortexBranchDivergence1` for details.
This analysis pass scans candidate branches and inserts reconvergence and divergence points by investigating loops and branches. The reconvergence and divergence points use Vortex intrinsics. 

## 5. Back-end Code Generation

To extend backend code generation, the Vortex ISA extensions and LLVM intrinsics are defined in `RISCVInstrInfoVX.td`. 
For example: 
- `VX_WSPAWN`: Spawn warps from a given address. 
- `VX_SPLIT`: Mark the beginning of a divergence region, pushes the current thread mask and related metadata onto the IPDOM stack, and returns the stacked address of the pushed state.
- `VX_JOIN`: Mark a reconvergence point, retrieving the saved state from the IPDOM stack to resume unified execution.
- `VX_PRED`: Supports predicated loops: it evaluates a loop predicate and updates the thread mask to reflect which threads should continue executing the next iteration. Once no threads satisfy the loop condition, `vx_pred` restores the original thread mask and jumps to the loop-exit block.
- `VX_BAR`: Vortex barrier instruction.
- `VX_MOV`: Vortex conditional move instruction.

In the final stage of VOLT divergence management, a machine-code optimization pass eliminates redundant register-copy instructions and corrects divergence split/join pairs to ensure a streamlined, divergence-safe binary. See `VortexBranchDivergence2` in `VortexBranchDivergence.cpp` for more details. 

Note that, since each Vortex thread has its own architectural register set (similar to a RISC-V CPU), VOLT can reuse LLVM-RISC-V register allocation and spilling without NVIDIA-style (register-usage-driven) occupancy constraints. In addition, VOLT uses LLVM’s `alloca` instruction to store IPDOM stack tags on the stack rather than in registers, thereby reducing register usage.
