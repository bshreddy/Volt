# Host Function Support Extension in CuPBoP


Volt extends CuPBoP with CUDA support for Vortex by leveraging CuPBoP’s original design principle: CUDA-facing behavior should be extended through modular runtime/ABI layers, rather than by modifying the core compiler pipeline. This separation keeps the kernel translation infrastructure reusable, while making it straightforward to introduce or refine host-runtime and memory semantics required by different open GPU architectures. In **CuPBoP-Vortex**, we apply this approach to preserve the original CUDA programming model (i.e., no source-level restructuring of CUDA applications) and lower CUDA runtime operations onto the open GPU architecture using Vortex intrinsics. The following section presents a concrete example of this design in practice by demonstrating how Volt extends host function support in the CuPBoP runtime.

---

## 1. `cudaMemcpy`: CUDA-like API preserved, Vortex-specific implementation underneath

A CUDA program typically issues a host-to-device copy as follows:

```c
// Application code (unchanged CUDA-style usage)
cudaMemcpy(d_x, x, N * sizeof(int), cudaMemcpyHostToDevice);
```

In CuPBoP-Vortex, the host runtime intercepts `cudaMemcpy(...)` and—when `cudaMemcpyKind == cudaMemcpyHostToDevice`—performs a two-step transfer using Vortex runtime primitives(`cupbop/runtime/src/vortex/host/cudaRuntimeImpl.cpp`):
	1.	Allocate a staging buffer (host-visible) and copy src into it.
	2.	Use a Vortex intrinsic (copy_to_dev) to transfer data from the staging buffer into the destination device address (dst).

```c	
// CuPBoP-Vortex host runtime (cupbop/runtime/src/vortex/host/cudaRuntimeImpl.cpp, implified)
cudaError_t cudaMemcpy(void *dst,
                       const void *src,
                       size_t count,
                       cudaMemcpyKind kind) {
  ...
  if (kind == cudaMemcpyHostToDevice) {
    // Create device context and allocate a staging buffer
    auto DC = DeviceContext::instance();
    auto staging_buf = DC->staging_alloc(count);
    auto host_ptr = staging_buf.data();

    // Copy host data into staging buffer
    memcpy((char*)host_ptr, src, count);

    // Destination is already a device pointer (e.g., from cudaMalloc)
    uint64_t mem_addr = (uint64_t)dst;

    // Copy staging buffer -> device destination using Vortex intrinsic
    DC->copy_to_dev(mem_addr, host_ptr, count);
  }
  ...
  return cudaSuccess;
}
```

Here, the application-level CUDA code remains unchanged; the CuPBoP-Vortex runtime lowers memory transfers to Vortex intrinsics under the hood. If you need to verify which Vortex device address a transfer targets, you can add a simple debug print in the host runtime, for example:

```c
printf("cudaMemcpyHostToDevice: src=%p, dst=%lu, count=%ld\n",
       src, mem_addr, count);
```

## 2. cudaMemcpyToSymbol vs. cudaMemcpy: what changes and why it matters

Another CUDA API whose host-side implementation must be specialized for different hardware backends is cudaMemcpyToSymbol. cudaMemcpyToSymbol differs from cudaMemcpy in two fundamental ways:

### 2.1 Symbol-based destination (name → address resolution)
- cudaMemcpy receives a device address as dst.
- cudaMemcpyToSymbol receives a symbol (e.g., `__device__` / `__constant__` global) and must resolve it to a device address.

### 2.2 Static global data segment vs. device heap allocation
- cudaMalloc allocations live in the device heap / dynamic allocation region.
- `__device__` / `__constant__` globals reside in the static device global data segment (global symbols, i.e., module .data/.bss-like storage).

In the CuPBoP-Vortex/Vortex execution model, the static global data segment gets reinitialized during kernel launch / device context setup, so writing directly to a symbol address too early can result in the value being overwritten by initialization logic.

```c
__constant__ float Data_d[3];

int main(int argc, char** argv) {
  ...
  float Data_h[3];
  Data_h[0] = 2.0f;

  cudaMemcpyToSymbol(Data_d, Data_h, 3 * sizeof(float));
  Kernel<<<128, 256>>>(A, B, C);
  ...
}
```

So given cudaMemcpytoSymbol usage shown above, CuPBoP-Vortex approach for cudaMemcpyToSymbol is to defer the final write until right before kernel launch

**Overview**

The host runtime implements cudaMemcpyToSymbol_host(...) as a two-phase mechanism:
1. Resolve the symbol name to its actual device global address (via a lookup).
2. Stage the payload into device memory (temporary allocation).
3. Defer the final copy into the static global data segment until immediately before kernel launch, using a metadata vector.
	

### 2.3. cudaMemcpyToSymbol_host(...): resolve symbol + stage payload + record a pending copy

The function below shows the relevant behavior:

- Find the symbol’s real device address (symbol_addr).
- Allocate temporary device memory (mem_addr) and copy src into it (via staging buffer + copy_to_dev).
- Record a pending “device-to-device” copy entry:
- dst: actual device global symbol address
- src: temporary device buffer address
- size: byte count
- offset: offset (currently only 0 supported)
	
```c
  // cupbop/runtime/src/vortex/host/cudaRuntimeImpl.cpp

	cudaError_t cudaMemcpyToSymbol_host(void *dst,
                                    const void *src,
                                    size_t count,
                                    size_t offset,
                                    cudaMemcpyKind kind,
                                    void *symbol) {

  // (1) Resolve the device global symbol address (symbol_addr)
  //     e.g., parse symbol name and lookup from a map/file generated at compile time
  uint64_t symbol_addr = /* ... resolved address ... */;

  // (2) Allocate temporary device memory and copy host payload into it
  auto DC = DeviceContext::instance();

  uint64_t mem_addr = 0; // temporary device buffer
  DC->dev_mem_alloc(count, VX_MEM_READ_WRITE, &mem_addr);

  // staging buffer for host->device transfer
  auto staging_buf = DC->staging_alloc(count);
  auto host_ptr = staging_buf.data();

  memcpy((char*)host_ptr, src, count);
  DC->copy_to_dev(mem_addr, host_ptr, count);

  // (3) Record a pending copy: (symbol_addr <- mem_addr)
  //     Store: [dst_symbol_addr, src_tmp_addr, size, offset]
  memcpy_symbol_single.push_back(symbol_addr);
  memcpy_symbol_single.push_back(mem_addr);
  memcpy_symbol_single.push_back((uint64_t)count);
  memcpy_symbol_single.push_back((uint64_t)offset);

  memcpy_symbol.push_back(memcpy_symbol_single);
  memcpy_symbol_single.clear();

  return cudaSuccess;
	}
```

### 2.4. Kernel wrapper: apply deferred symbol updates right before kernel launch

Inside the kernel wrapper (`cupbop/examples/vortrex_debug/kernel_wrapper.cpp`, right before `vx_spawn_threads(...)`), CuPBoP-Vortex consumes the recorded metadata and performs the final device-side copy so that the kernel observes the updated symbol values.

Conceptually:
- For each pending entry: dst_symbol_addr <- src_tmp_addr of size bytes
- Then launch the kernel (vx_spawn_threads)
	
```c
// cupbop/examples/vortrex_debug/kernel_wrapper.cpp (simplified)

int main() {
  ...

  // Apply deferred cudaMemcpyToSymbol updates immediately before kernel launch
  for (size_t i = 0; i < memcpy_symbol.size(); i++) {
    uint64_t dst_symbol_addr = memcpy_symbol[i][0];
    uint64_t src_tmp_addr    = memcpy_symbol[i][1];
    size_t   size            = (size_t)memcpy_symbol[i][2];
    size_t   offset          = (size_t)memcpy_symbol[i][3];

    // Perform device-to-device copy into the static global data segment.
    DC->copy_dev_to_dev(dst_symbol_addr, src_tmp_addr, size);
  }

  // Launch kernel
  return vx_spawn_threads(...);
}
```	
To summarize the cudaMemcpyToSymbol implementation and the extension point in Volt:
- Resolve the symbol name to the real device address.
- Stage the payload into temporary device memory.
- Defer the final device-to-device copy until immediately before kernel launch, avoiding conflicts with static global initialization behavior.
- Optionally, after kernel completion, copy back only when the symbol is writable (`__device__`) and write-back semantics are required.

This design allows existing CUDA applications to run without source changes, while adapting memory semantics to Vortex’s open/reconfigurable GPU execution stack.

## 3. Adding Custom CUDA Runtime API Extensions

As with `cudaMemcpy` and `cudaMemcpyToSymbol`, users may want to add additional CUDA host-runtime APIs or define custom host APIs that can be called directly from CUDA-facing code. In Volt (CuPBoP-Vortex), this can be done by extending the host runtime implementation and header declarations.

First, add a new host-side API implementation in `cupbop/runtime/src/vortex/host/cudaRuntimeImpl.cpp`:
```c
// cupbop/runtime/src/vortex/host/cudaRuntimeImpl.cpp
cudaError_t foo(void *A, const void *B) {
  ...
}
```

Then, declare the corresponding function prototype in `cupbop/runtime/include/cudaRuntimeImpl.h`:
```c
// cupbop/runtime/include/cudaRuntimeImpl.h
cudaError_t foo(void *A, const void *B);
```

Once the implementation and header are added, the new API can be invoked directly from CUDA source code without requiring source-level restructuring.

If the newly added host API introduces behavior that must be reflected at kernel-launch time (for example, requiring a pre-launch copy, post-launch synchronization, or additional metadata propagation), you can also extend the wrapper-generation logic in `cupbop/compilation/KernelTranslation/src/generate_wrapper.cpp`:

Specifically, add the necessary logic inside the function `create_kernel_wrapper_function`:
```c
// cupbop/compilation/KernelTranslation/src/generate_wrapper.cpp
void create_kernel_wrapper_function(llvm::Module *M) {
  ...
}
```

This wrapper-generation path uses a stringstream-based workflow to collect kernel metadata and emit the corresponding kernel_wrapper.cpp, which is then compiled together with the IR-translated kernel.bc during the final build. After these updates, the newly defined host API becomes available to CUDA applications and can be called directly from user code, for example:
```c
// User CUDA source code
int main(int argc, char** argv) {
  ...
  foo(A, B);
  ...
}
```

Overall, Volt is designed with host-side API extensibility in mind, so that users can introduce CUDA-facing runtime extensions and have them apply directly to existing CUDA applications.
