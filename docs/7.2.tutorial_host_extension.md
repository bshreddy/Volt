# Host Function Support Extension in CuPBoP

CuPBoP is designed so that CUDA-facing behavior can be extended primarily in the host–runtime layer, rather than by modifying the core compiler pipeline. This modular host–runtime design makes it straightforward to introduce or refine runtime semantics and memory behaviors needed by different open GPU architectures, while keeping the kernel translation infrastructure reusable. 

In this tutorial, we use shared-memory support as a concrete example to show how such extensions are implemented, and how users can extend the selection and validation workflow for alternative behaviors with minimal disruption to existing CUDA applications. CUDA programs typically assume a specific shared-memory abstraction: a per-CTA scratchpad with fast, low-contention access. On open GPU targets, however, the best way to realize this abstraction is not unique and often changes with the underlying architecture. For example, current Vortex GPU does not provide a dedicated shared-memory. As a result, CUDA shared memory must be reinterpreted under the target memory model, and the implementation can becomes a policy choice rather than a fixed rule.

CuPBoP therefore treats shared memory as a lowering strategy that can be swapped depending on the kernel’s access pattern and the available memory hierarchy. In some cases, casting shared memory to per-core local memory can reduce contention and improve locality. In other cases, mapping shared data to a global allocation can improve cache hit rates and deliver better throughput. CuPBoP makes these trade-offs easy to explore by allowing users to switch shared-memory mapping policies inside the kernel translation passes.

In this tutorial, we walk through two shared-memory lowering strategies—shared-to-global casting and shared-to-local casting—and show how to modify the mapping policy when needed.

#### 1. Extending Kernel Translator in CuPBoP_Vortex
Inside `KernelTration/src/memory_hierarchy.cpp` we have two functions, 
```c
void mem_share2global_sche_2(llvm::Module *M);
void mem_share2local(llvm::Module *M);
``` 

At a high level, both functions iterate over module-level global variables and identify shared-memory globals by checking whether addressSpace == 3. The key difference is what memory object they replace the shared global with, and how they ensure per-CTA isolation.

First, mem_share2global_sche_2 replaces each shared global with a newly created module global variable in address space 1. As shown below, it allocates a replacement global in AS=1 and then rewrites uses so that each CTA indexes into its own slice. The CTA identifier (block ID) is loaded at runtime, ensuring that different CUDA blocks access disjoint regions of the new global object.

```c
auto createNoTLSGlobal = [&](Type *Ty, StringRef Name) -> GlobalVariable* {
  auto *Init = isa<ArrayType>(Ty) ? (Constant*)ConstantAggregateZero::get(Ty)
                                  : (Constant*)Constant::getNullValue(Ty);
  auto *GV = new GlobalVariable(
      *M, Ty, /*isConst=*/false,
      GlobalValue::ExternalLinkage, Init, Name,
      /*InsertBefore=*/nullptr,
      GlobalValue::NotThreadLocal, /*AddressSpace=*/1);
  GV->setThreadLocal(false);
  GV->setThreadLocalMode(GlobalVariable::NotThreadLocal);
  return GV;
};
```

In contrast, mem_share2local maps shared memory to Vortex per-core local memory. When a shared-memory variable is detected, the transformation inserts runtime allocation logic so that each CTA’s leader thread (e.g., tid == 0) allocates a local-memory region using the Vortex local-memory allocation intrinsic. The pass then rewrites shared-memory accesses to use the allocated base address, enabling accesses to be served from Vortex local memory.

```c
FunctionCallee vxAlloc = M->getOrInsertFunction(
    "vx_local_alloc", FunctionType::get(I8Ptr, {I32}, false));
```

These memory-casting transformations are invoked from the init_block function in `KernelTranslation/src/init.cpp`. If a user changes the call site to select either `mem_share2global_sche_2` or `mem_share2local`, they can evaluate the two shared-memory mappings without modifying the CUDA source code. This makes it straightforward to experiment with alternative memory hierarchies and validate which mapping for given hardware configuration is more effective for the kernel.
